# Dynamo 3.x `.dyn` JSON schema

Source of truth for emitting `.dyn` files compatible with Revit 2026 / Dynamo 3.6. All examples here are extracted from working SOFiSTiK example graphs.

## Top-level structure

```json
{
  "Uuid": "<new GUID>",
  "IsCustomNode": false,
  "Description": "<one-line description>",
  "Name": "<script name, used as window title>",
  "ElementResolver": { "ResolutionMap": {} },
  "Inputs":  [ /* declarations of Player input nodes — see "Inputs / Outputs declarations" below */ ],
  "Outputs": [ /* declarations of Player output nodes (Watch nodes marked IsSetAsOutput) */ ],
  "Nodes": [ /* the actual nodes */ ],
  "Connectors": [ /* wires between node ports */ ],
  "Dependencies": [],
  "NodeLibraryDependencies": [ /* package + external file deps, see below */ ],
  "EnableLegacyPolyCurveBehavior": true,
  "Thumbnail": "",
  "GraphDocumentationURL": null,
  "ExtensionWorkspaceData": [
    { "ExtensionGuid": "28992e1d-abb9-417f-8b1b-05e053bee670", "Name": "Properties",        "Version": "3.3", "Data": {} },
    { "ExtensionGuid": "DFBD9CC0-DB40-457A-939E-8C8555555A9D", "Name": "Generative Design", "Version": "8.2", "Data": {} }
  ],
  "Author": "",
  "Linting": {
    "activeLinter": "None",
    "activeLinterId": "7b75fb44-43fd-4631-a878-29f4d5d8399a",
    "warningCount": 0, "errorCount": 0
  },
  "Bindings": [],
  "View": { /* see "View block" below */ }
}
```

All `Id` fields throughout the file are **lowercase 32-char hex strings without dashes** (e.g. `c40cc5712c1e4207b21f9478600af92a`). One per node, one per input port, one per output port, one per connector. They must be globally unique within the file. Use `uuid.uuid4().hex` to generate.

## Node entries (`Nodes` array)

Every node has these common fields:

```json
{
  "ConcreteType": "<type-specific, see below>",
  "Id": "<32-hex node id>",
  "NodeType": "<see below>",
  "Inputs":  [ { "Id": "<32-hex port id>", "Name": "<port name>", "Description": "...",
                 "UsingDefaultValue": false, "Level": 2, "UseLevels": false, "KeepListStructure": false }, ... ],
  "Outputs": [ { "Id": "<32-hex port id>", "Name": "<port name>", "Description": "...",
                 "UsingDefaultValue": false, "Level": 2, "UseLevels": false, "KeepListStructure": false }, ... ],
  "Replication": "Auto" | "Disabled",
  "Description": "<tooltip>"
}
```

Plus type-specific fields. The five types you'll use:

### 1. Zero-Touch function call (`DSFunction`)

For any call into a .NET assembly: SOFiSTiK package, Revit Nodes (`Element.SetParameterByName`, `Parameter.ParameterByName`, `Parameter.Value`), DSCore (`List.Join`).

```json
{
  "ConcreteType": "Dynamo.Graph.Nodes.ZeroTouch.DSFunction, DynamoCore",
  "Id": "...",
  "NodeType": "FunctionNode",
  "Inputs": [ ... ports matching the function signature ... ],
  "Outputs": [ ... usually one port ... ],
  "FunctionSignature": "Namespace.Type.Method@Arg1Type,Arg2Type",
  "Replication": "Auto",
  "Description": "<auto-generated, e.g. 'Returns ... \\n\\nMethod (a: T, b: T): R'>"
}
```

`FunctionSignature` examples (verified):
- `Revit.Elements.Parameter.ParameterByName@Revit.Elements.Element,string`
- `Revit.Elements.Parameter.Value`  *(no `@...` because no args)*
- `Revit.Elements.Element.SetParameterByName@string,var`
- `SOFiSTiK.Analysis.Utils.GetPhysicalCounterpart@Revit.Elements.Element[]`

For variable-argument functions (e.g. `List.Join`), use `DSVarArgFunction`:

```json
{
  "ConcreteType": "Dynamo.Graph.Nodes.ZeroTouch.DSVarArgFunction, DynamoCore",
  "FunctionSignature": "DSCore.List.Join@var[]..[]",
  "FunctionType": "VariableArgument",
  ... (rest as above; Inputs has list0, list1, list2... as needed)
}
```

### 2. String input

```json
{
  "ConcreteType": "CoreNodeModels.Input.StringInput, CoreNodeModels",
  "Id": "...",
  "NodeType": "StringInputNode",
  "Inputs": [],
  "Outputs": [ { "Id": "...", "Name": "", "Description": "String", "UsingDefaultValue": false,
                 "Level": 2, "UseLevels": false, "KeepListStructure": false } ],
  "Replication": "Disabled",
  "Description": "Creates a string",
  "InputValue": "<the actual string>"
}
```

### 3. File path input

```json
{
  "ConcreteType": "CoreNodeModels.Input.Filename, CoreNodeModels",
  "Id": "...",
  "NodeType": "ExtensionNode",
  "Inputs": [],
  "Outputs": [ { "Id": "...", "Name": "", "Description": "File path",
                 "UsingDefaultValue": false, "Level": 2, "UseLevels": false, "KeepListStructure": false } ],
  "Replication": "Disabled",
  "Description": "Allows you to select a file on the system and returns its file path",
  "HintPath": "C:\\absolute\\fallback\\path\\file.ext",
  "InputValue": "..\\..\\relative\\path\\file.ext"
}
```

`HintPath` is the absolute fallback. `InputValue` is the relative path stored in the file. For new graphs, set both to an empty string `""` if the user hasn't given a path.

### 4. Python Script (CPython3)

```json
{
  "ConcreteType": "PythonNodeModels.PythonNode, PythonNodeModels",
  "Code": "<full python code as a single JSON string with \\r\\n line breaks>",
  "Engine": "CPython3",
  "VariableInputPorts": true,
  "Id": "...",
  "NodeType": "PythonScriptNode",
  "Inputs": [
    { "Id": "...", "Name": "IN[0]", "Description": "Input #0", "UsingDefaultValue": false,
      "Level": 2, "UseLevels": false, "KeepListStructure": false }
    /* one entry per IN[i] used in the code */
  ],
  "Outputs": [
    { "Id": "...", "Name": "OUT", "Description": "Result of the python script", "UsingDefaultValue": false,
      "Level": 2, "UseLevels": false, "KeepListStructure": false }
  ],
  "Replication": "Disabled",
  "Description": "Runs an embedded Python script."
}
```

Inside `Code`, use `\r\n` line endings and JSON-escape quotes/backslashes. The builder handles this for you.

### 5. Watch (output display)

```json
{
  "ConcreteType": "CoreNodeModels.Watch, CoreNodeModels",
  "WatchWidth": 200.0,
  "WatchHeight": 200.0,
  "Id": "...",
  "NodeType": "ExtensionNode",
  "Inputs":  [ { "Id": "...", "Name": "", "Description": "Node to show output from", ... } ],
  "Outputs": [ { "Id": "...", "Name": "", "Description": "Node output", ... } ],
  "Replication": "Disabled",
  "Description": "Visualizes a node's output"
}
```

### Other useful UI nodes

- **All Elements of Category** — `"DSRevitNodesUI.ElementsOfCategory, DSRevitNodesUI"`, NodeType `ExtensionNode`. Input port `Category`, output `Elements`.
- **Categories dropdown** — `"DSRevitNodesUI.Categories, DSRevitNodesUI"`, NodeType `ExtensionNode`. Has `SelectedIndex` (int) and `SelectedString` (e.g. `OST_AnalyticalMember`, `OST_AnalyticalPanel`, `OST_StructuralFraming`, `OST_StructuralColumns`, `OST_Walls`, `OST_Floors`).
- **Select Model Elements By Category** — `"Dynamo.ComboNodes.DSModelElementsByCategorySelection, DSRevitNodesUI"`. Captures a manual user selection. Has `SelectedIndex`, `SelectedString`, and `InstanceId` (array of element unique-id strings).
- **Boolean** — `"CoreNodeModels.Input.BoolSelector, CoreNodeModels"`, NodeType `BooleanInputNode`, with `InputValue: true|false`.
- **Integer slider** — `"CoreNodeModels.Input.IntegerSlider, CoreNodeModels"`, with `InputValue`, `MaximumValue`, `MinimumValue`, `StepValue`.

## Connectors

```json
{
  "Start": "<output port id of source node>",
  "End":   "<input  port id of target node>",
  "Id":    "<32-hex>",
  "IsHidden": "False"
}
```

Note `IsHidden` is the **string** `"False"`, not the boolean. Match the source data exactly.

## NodeLibraryDependencies

Required when the graph uses package nodes — Revit will warn if missing. For SOFiSTiK:

```json
{
  "Name": "SOFiSTiK Analysis + Design",
  "Version": "2026.5.0",
  "ReferenceType": "Package",
  "Nodes": [ "<id of every SOFiSTiK node in the graph>" ]
}
```

For external file references (Excel, .cdb), one entry per file:

```json
{
  "Name": "main.xlsx",
  "ReferenceType": "External",
  "Nodes": [ "<id of the Filename node referencing it>" ]
}
```

## View block

```json
"View": {
  "Dynamo": {
    "ScaleFactor": 1.0,
    "HasRunWithoutCrash": true,
    "IsVisibleInDynamoLibrary": true,
    "Version": "3.2.2.5494",
    "RunType": "Manual",
    "RunPeriod": "1000"
  },
  "Camera": { "Name": "_Background Preview",
              "EyeX": -17.0, "EyeY": 24.0, "EyeZ": 50.0,
              "LookX": 12.0, "LookY": -13.0, "LookZ": -58.0,
              "UpX": 0.0, "UpY": 1.0, "UpZ": 0.0 },
  "ConnectorPins": [],
  "NodeViews": [ /* one per node — see below */ ],
  "Annotations": [ /* groups */ ],
  "X": 100.0, "Y": 400.0, "Zoom": 0.5
}
```

Note: emitting `Version: "3.2.2.5494"` works in Dynamo 3.6 — files made in 3.2 open and run in 3.6 fine. Don't bump it unless you have a 3.6-only feature.

### NodeViews (one per node, **mandatory**)

```json
{
  "Id": "<same node id>",
  "Name": "<display name shown above the node>",
  "IsSetAsInput": false,
  "IsSetAsOutput": false,
  "Excluded": false,
  "ShowGeometry": true,
  "X": <number>, "Y": <number>
}
```

The node will not render without a NodeView. The `Name` for `StringInput` and `Filename` nodes when marked as Player input becomes the **label shown in the Dynamo Player** — make it user-facing (e.g. `"Path to .cdb"`, not `"File Path"`).

`Excluded: true` greys-out the node ("freeze" — node and downstream skipped during run). Use it for "Alternative input" branches you want present-but-disabled.

### Marking Player inputs and outputs

Two coordinated changes are required for **each** Player-visible parameter:

1. In `View.NodeViews[].IsSetAsInput`: set `true` on the input node (StringInput, Filename, BoolSelector, IntegerSlider, …). Also set the user-facing `Name`.
2. In top-level `Inputs[]`: append a declaration:
   ```json
   { "Id": "<node id>",
     "Name": "<same as NodeView Name>",
     "Type":  "string"  /* or "boolean", "number", "filename" — match the source node */,
     "Type2": "string",
     "Value": "<current value, same as InputValue>",
     "Description": "<tooltip>" }
   ```

For Player **outputs** (Watch nodes):

1. `View.NodeViews[].IsSetAsOutput: true`, and a meaningful `Name` (e.g. `"Exported LoadCases"`).
2. Top-level `Outputs[]` declaration:
   ```json
   { "Id": "<watch node id>",
     "Name": "<same as NodeView Name>",
     "Type": "unknown",
     "InitialValue": "",
     "Description": "Visualizes a node's output" }
   ```

If a node is **not** a Player input/output, both NodeView flags stay `false` and there is **no** entry in the top-level `Inputs`/`Outputs` arrays.

### Annotations (groups)

```json
{
  "Id": "<32-hex>",
  "Title": "Input: select analytical elements",
  "DescriptionText": "",
  "IsExpanded": true,
  "WidthAdjustment": 0.0,
  "HeightAdjustment": 0.0,
  "Nodes": [ "<id>", "<id>", ... ],
  "HasNestedGroups": false,
  "Left": <x of bounding box>,
  "Top":  <y of bounding box>,
  "Width":  <px>,
  "Height": <px>,
  "FontSize": 36.0,
  "GroupStyleId": "00000000-0000-0000-0000-000000000000",
  "InitialTop":    <number>,
  "InitialHeight": <number>,
  "TextblockHeight": 63.333333333333336,
  "Background": "#FFC1D676"
}
```

**Recommended group conventions for this skill:**

| Purpose            | Background  |
|--------------------|-------------|
| Input group        | `#FFC1D676` (green) |
| Processing group   | `#FFB9F9E1` (mint)  |
| Output group       | `#FFFFC999` (orange)|
| SOFiSTiK callout   | `#FF48B9FF` (blue)  |

The builder computes `Left/Top/Width/Height` from the contained nodes' coordinates with a margin.

## Layout convention

For graphs built by this skill, lay out left-to-right by stage:

- **Inputs** group: X ≈ -600 to -200
- **Processing** group: X ≈ 0 to 1500 (further right per pipeline depth)
- **Outputs** group: X ≈ 1700+

Vertical spacing ~150 px between sibling nodes; horizontal stride ~350 px between processing stages. The builder defaults to these.
